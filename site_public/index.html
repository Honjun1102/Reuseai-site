LLM inference-side compute avoidance

Response Cache / Reuse System

A request-level cache / replay layer for repeat-heavy LLM traffic. Lower token cost, reduce TTFT, and keep throughput high without changing the model.

Problem

Repeat requests still pay full inference cost.

Support macros, API retries, parameterized requests, and workflow prompts repeat constantly. Every duplicate request burns tokens, adds latency, and consumes backend capacity.

  • Exact-match replay avoids re-running the same request.
  • Cache eligibility keeps behavior predictable.
  • Private API and workflow traffic show the clearest reuse patterns.
Solution

Drop-in request-level cache / replay / reuse.

Fingerprint

Build an exact-match cache key from model, prompt, system prompt, and request params.

Replay

Return cached responses immediately when the same request repeats.

Eligibility

Use cacheability policy, TTL, and invalidation to keep behavior predictable.

Deployment

Proxy-first, self-hosted, no model changes.

  • Standalone proxy in front of Ollama-compatible inference services.
  • Minimal config integration: backend URL, model, TTL.
  • Exact-match cache, replay, TTL, invalidation, and metrics included.

Client → response cache proxy → Ollama / vLLM / PyTorch-backed API

This is an inference-side compute avoidance layer, not a model rewrite or kernel optimization project.
